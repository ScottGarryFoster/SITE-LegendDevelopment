Public Release / Licences

How we develop and release

Some applications developed for Fated.Quest are to be publicly available for anyone to download or redistribute as they see fit. Public over private development requires a certain level of process which must be followed to ensure that public code is always secure and safe. If an application is distributed publicly this puts an onus on us to ensure quality control.

  1. Before a public release (even a beta) the repo is private to ensure nobody downloads broken code
  2. During a public release development takes place on separate branches and is merged into the main branch
    • The exception being code which is developed a private and moved to a public for security reasons
  3. All public releases go through our QA program before an actual release is made

Development and Engagement

Development uses a ticket system which relate to the commits. Each development feature and release requires a branch. It is largely up to the team and developer for the branch layout (each project is different) so long as live does not occur on main.

All commits to main must be:

  1. Internally tested and QA Tested - use the testing process to ensure the release is safe
  2. Change Listed - Use the change list standard to update the documentation and keep everything neat
  3. Released built - Follow-up commits to main by a release built where appropriate (packaged) to allow people to download the latest version

Engagement on project takes place via the ticket system for bugs and tasks. For public external requests these should be triaged with project leaders and taken into account the internal use of the development.

All engagements but especially public engagements must be:
