Text Display

Introduction

Informs the user about the world, actions, and story via the medium of characters on a screen. Lays out the text in such a way that any action may be given and for custom characters.

Content

1

There is 64 characters across and 36 lines max on the screen in a Monospaced manner using the font Retro Gaming. https://www.dafont.com/retro-gaming.font

2

Split the screen into three uneven sections. Left and right containing 13 tiles and the centre 38 tiles. Left is section A, Middle Section B, Right section C. The module allows separate text in any section.

3 Any alphanumeric character may be displayed along with basic punctuation found on a keyboard. Custom characters may be entered using non-standard lettering in languages outside of English. For instance, Ã© might become a heart symbol defined custom.
4 Custom characters are defined by a designer with the images for characters provided.
5 Any colour may be applied to the text and the colour may be applied mid-sentence for emphasis.
6 Upon entry of text the text may shift upwards (bottom append), replace or shift downwards.
7 Pauses in text entry may be inserted of varying lengths to allow for varied text speed. Text speed in general should be customisable.
8 Text should write on screen from left to right with a set delay for each character (outside of custom delays).
9 Text should be able to trigger musical queues or events outside of the module. This should be kept templated as there are likely many systems to interact with (event based).
10

In section B text may be entered by the player at the bottom. This text is displayed in a format chosen by designers in section B shift upwards. The text should send events for other systems to react. See Text Entry Design, for instructions on this control.

11 The background of every section and every character may be customised when sending information to the system.
12 Text should be replaceable per line. To allow for partial refresh of information, interaction with a single line should be possible for example changing line 7 to a new line of text. Any character in any line should be replaceable.
13 The state of all sections and content should be save and loadable with the text in all sections saved including entry such that if the module were reloaded the state may be returned to the original state.
14 Any character may contain animation data affecting the character image, colour, or background colour.
15 Colours are defined by the Hex code and any colour may be used for any item on the screen.

Why

The game is a text adventure, so text display is a key element of this. It allows full customisation of characters, how the display and how interactions take place. The flexibility to full customise anything mean that future design may take advantage of these systems.

A higher flexibility with this text display means better future designs and better story telling which are the backbone of the genre.

A monospaced font portrays two key elements, retro gaming with text adventures and allows for ASCII art as using a set spacing mean vertical lines are possible.

Animations will become useful with a battle system.

Music and event queues will become useful to bring people into the experience and needs to be in this stage as text speed is customisable therefore it should be tied to this system.

Custom characters allow us to draw characters on the screen for a battle system.

Pausing mid-sentence should allow for the storyteller to keep information away from the player and build suspense.

Background Research

Other text adventures such as the ones produced through TextAdventures.com using Quest and Squiffy use a column of text to portray information to the user. The left and right margins should allow for future implementations of inventory, map, and battle systems these engines also use.

Clicking on items in the text is a key element of Quest, Squiffy and Twine which is a reason for that option in our engine.

The text entry is found within Quest and allows for a broader window from the player into the game. This is found in older adventure games such as Zork.

Considerations

Game Rating

Text display itself does not have the ability to raise the rating depending on the custom characters. The ratings are based on content more than systems.

Patents, Trademarks and Copyright

This likely does not overlap with or qualify for a patent as it is essentially a word processor.

Build and Systems

Taking the characters and text would become a build process from the tools. This would need to be within the game executable.

Rendering the element on screen would be a required system.

Tools

There needs to be a tool to create characters.

When eventually the input for text system is created a tool should be created for this which allows for text with these elements.

Save file implications

The state of the screen would need to be included in the save file with loading included.

Performance Implications

This would increase the performance requirements but not by much. It is just a word processor, but the implementation is likely to be such to use images. This is likely to increase the performance therefore but not by much and is likely a necessity.

Effect on other systems

This would be the system a lot of other systems interact with. As the output to the user the economy, battle and map systems will all interact with this.

This therefore needs to be compatible with other systems and be flexible enough for those systems.
